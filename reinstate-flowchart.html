Analysis based on 179 PSR tickets (Jan 2025–Apr 2026). ~85 PSRs estimated addressable by the three planned stories. Current flow confirmed from SFL_Reinstate_Subscription.flow-meta.xml and reinstate-subscription.handler.ts.

Issue Breakdown & Recommended Fixes

# Issue Est. PSRs What current code does (or doesn't do) Fix Ticket Status
1 Cross-month Dunning cases blocked / generic code:null
Rep gets useless error or told to "set up as winback" for a Dunning debt case. 		~50 Flow checks isWithinCurrentMonth on serviceCutOffDate. If cutoff date is from cancellation month (not current month), even debt cases are redirected to winback. On failure, shows {apexaction…message} which returns "null code:null". No Slack alert raised. Fix cross-month check: allow debt cases regardless of cutoff month, auto-adjust cutoff date. Add human-readable error + Slack alert on failure. BTE-3501 + spike ⚠ Partially scoped
2 Dunning re-cancels account the night after reinstatement
Dunning sees unpaid invoices >80 days and re-cancels. 		~30 Billing handler (reinstate-subscription.handler.ts) has no pre-validation. ADR020 (Feb 2025) says billing should validate 80-day rule + payment schedule before reinstating — decision agreed but never implemented. Add pre-check in ScreenFlow: if >80 days + no payment received on oldest invoice → block. Even partial payment unblocks. Payment plan alone not sufficient. BTE-3509 ❌ Unassigned
3 Vague PSRs — no diagnostic detail ~30 Failure screen shows {message} from API. API returns "null code:null". Rep has nothing useful to paste into PSR description. Fix error message (BTE-3501 alerting). Also: add required error message field to PSR form. No ticket for PSR form Not scoped
4 Self-ascension reversals (separate process) ~20 Not a reinstate bug. Different process under BTE-4087. Track separately. BTE-4087 🔀 Separate track
5 Debt paid but reinstate still blocked
DunningHold flag not cleared after $0 balance. 		~18 No $0 balance check in ScreenFlow. DunningHold__c not auto-cleared. Flow just proceeds to API and the API blocks it. If balance = $0 → auto-set Collection Case = "Paid" and allow reinstate. BTE-3501 ❌ Unassigned
6 processedThroughDate — unrecoverable
Second reinstate attempt after Dunning loop creates sub with no invoices. 		~10 Billing handler throws BadRequestException when processedThroughDate is null. No fallback. Billing service should compute from last invoice period or default to current date. New spike ❌ Not scoped
7 DC flag not cleared after reinstatement
Collection Status remains "Approved to Send to DC" → Dunning re-triggers. 		~7 Success screen just shows ✅ message. No Collection Case record is updated by the ScreenFlow after successful reinstate. After success, auto-update Collection Case to "Paid" or "Paid from DC". BTE-3020 ❌ Unassigned (19 months)
8 Admin/Zuora billing mismatch ~8 ScreenFlow has no path for accounts where Zuora sub is missing. PSR-4648 added a self-heal in the cancellation handler but doesn't fix existing broken accounts. Separate investigation. PSR-4648 mitigates future occurrences. No ticket Partial

Edge Cases to Handle

Scenario Expected Behaviour Risk if Not Handled
Rep reinstates → Dunning cancels same night BTE-3020 auto-updates Collection Case to "Paid" → prevents re-cancellation Loop continues even after BTE-3509/3501 ship
Multiple invoices — oldest paid, others remain Check oldest invoice date, not total balance Risk of blocking legitimate reinstates if total balance logic used
Payment plan exists but no actual payment made Payment plan alone does NOT qualify — at least one payment (even partial) must have been applied to the >80-day invoice Allowing reinstate on plan alone → Dunning re-cancels overnight
Rep manually changes Collection Status in Salesforce (bypasses ScreenFlow) BTE-3020 auto-corrects post-reinstate regardless of path Without BTE-3020, Dunning still re-triggers even after BTE-3509/3501 ship
Account reinstated multiple times in same month Each reinstate creates new sub with no invoices → Dunning re-triggers Need reinstate date check in Dunning logic before re-cancelling
DC case with partial payment — not $0 but under 80 days Allow reinstate if oldest invoice < 80 days Partial payment scenarios unclear — needs Rev Ops confirmation
Account manually cancelled by rep (not via Dunning) May not be eligible for ScreenFlow reinstate path Manually cancelled accounts need a separate handling path
Account reinstated months after cancellation — start date backdated Start date = reinstatement date, not original ChargedThroughDate Tradie billed for inactive months; manual invoice write-off required (PSR-4834)
Cancelled sub has null processedThroughDate Billing computes from last invoice period or defaults to current date Billing throws BadRequestException; account unrecoverable via UI (PSR-4268)
Legacy PackageType__c value no longer in catalog Billing falls back to rate plan name for package type Fails with "invalid multi-select option value" — data fix only, no code guard (PSR-4796)

Key Correction vs. Original Diagram

What the original diagram assumed What the code actually shows
"No pre-checks, no validation — direct API call" Flow has permission check, active-sub check, serviceCutOffDate / cancellationReason checks, and a subscription detail confirmation screen before calling the API
Cross-month cases silently fail with code:null Most cross-month cases are explicitly BLOCKED by the flow with "please set it up as a winback" — including cross-month Dunning debt cases (the bug)
"Dunning re-cancels after every reinstate" Only reinstates that pass the flow's checks AND happen to be in the same calendar month as cancellation proceed to the API. Cross-month debt cases never even reach the API.
Billing handler has no pre-checks (correct) Confirmed — reinstate-subscription.handler.ts runs activation workflow directly with no debt/dunning validation
